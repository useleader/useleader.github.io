# 00-学习地图

## 一句话主线

YASA-VulnAgent 的技术链路是：源码先被 YASA 转成 UAST 和结构化程序事实，程序分析模块给出数据流/污点路径，LLM 只在这些事实约束下判断可利用性，再通过 PoC 沙箱验证和报告生成闭环降低误报。

## 五层知识地图

| 层次 | 要学什么 | 在项目中的作用 | 对应文件 |
|---|---|---|---|
| 表示层 | AST、CFG、DFG、PDG、SSA、CPG、UAST | 把代码转为可分析结构 | [[02-程序表示与UAST]] |
| 数据流层 | points-to、alias、flow/context/field/path sensitivity | 判断值和对象如何传播 | [[03-指针别名与数据流分析]] |
| 污点层 | source、sink、sanitizer、propagator、filter、taint path | 形成漏洞证据链 | [[04-污点分析]] |
| 查询层 | CodeQL、Datalog、伪 UQL、checker | 把漏洞模式写成规则 | [[05-查询语言与漏洞规则]] |
| Agent 层 | MCP tool、SkillResult、PEV、grounding | 让 LLM 基于程序事实推理 | [[06-YASA与Agent集成]] |

## 关键误区

1. UAST 不是漏洞检测结果，它是多语言统一表示。
2. 污点分析不是 LLM 猜漏洞，而是 source-to-sink 路径求解。
3. sanitizer 不是“出现过滤函数就安全”，要看路径、上下文和实际语义。
4. YASA-MCP 的具体接口在官方材料确认前不能当作确定 API。
5. 申报书创新点不能停留在“YASA + Agent”，必须落到结构化事实 grounding、UAST 多语言统一、PEV 验证闭环或评估方案。

## 学习验收问题

1. 我能否用 5 分钟解释 YASA-VulnAgent 的完整链路？
2. 我能否手工标出一个 SQLi 例子的 source、propagator、sanitizer、sink？
3. 我能否说明 points-to/alias 为什么会影响污点分析误报？
4. 我能否写出一条 SQLi 伪查询？
5. 我能否设计一个 `query_taint_paths` 风格工具的输入输出草案，并标注它是推断设计？


## 逐条答疑：我到底应该学什么

结论先放前面：你不需要把“编译原理、程序分析、CodeQL、LLM Agent”全部从头学到专家水平。对 YASA-VulnAgent 来说，最重要的是学会一条主线：

```text
代码如何被表示
-> 值和对象如何传播
-> 不可信输入如何流向危险操作
-> 漏洞模式如何写成规则
-> Agent 如何调用这些结构化事实做判断和验证
```

所以学习时不要把每个术语孤立记忆。你可以不断问同一个问题：**这个概念帮助系统回答了什么问题？** 比如 AST 回答“代码长什么样”，CFG 回答“程序可能怎么走”，DFG 回答“值怎么传”，points-to 回答“引用指向谁”，污点分析回答“不可信数据是否到达危险操作”，查询语言回答“我要找哪类路径”。

## 第一层：表示层是在给代码建模，不是在直接找漏洞

表示层的任务是把源代码转换为机器能分析的结构。它不是漏洞检测结果，也不是最终证据，只是后续分析的地基。

你可以把表示层理解成“代码地图”。地图越清楚，后续找路径越容易；但地图本身不会告诉你“这里一定有漏洞”。

### AST：代码的语法树

AST 是 Abstract Syntax Tree，抽象语法树。它把源码拆成树状结构，表达“这段代码由哪些语法成分组成”。

比如：

```python
sql = "select * from users where name = '" + name + "'"
```

在 AST 里大致会被表示成：

```text
Assignment
  left: Identifier(sql)
  right: BinaryExpression(+)
    left: StringLiteral("select ...")
    right: Identifier(name)
```

AST 能回答：

1. 这里是不是赋值语句？
2. 左边变量是谁？
3. 右边是不是字符串拼接？
4. 这个函数调用的名字是什么？

AST 不能直接回答：

1. `name` 是否来自用户输入？
2. `sql` 后面有没有被执行？
3. 这条路径是否跨函数传播？

所以 AST 是第一步，它告诉分析器“代码长什么样”，但还没有完整的数据传播语义。

### CFG：程序可能怎么执行

CFG 是 Control Flow Graph，控制流图。它表示程序执行顺序、分支、循环、异常等控制路径。

比如：

```python
if is_admin:
    sql = raw_sql
else:
    sql = safe_sql
db.execute(sql)
```

CFG 关注的是：

```text
进入函数
  -> 判断 is_admin
    -> true 分支: sql = raw_sql
    -> false 分支: sql = safe_sql
  -> db.execute(sql)
  -> 退出函数
```

这里的“流”不是数据流，而是控制流，也就是程序执行可能经过哪些语句。

它对漏洞分析很重要，因为很多漏洞是否成立取决于分支条件。例如：

```python
if is_safe(url):
    requests.get(url)
```

如果路径敏感分析能理解 `requests.get(url)` 只在 `is_safe(url)` 成立后执行，那么它可能降低 SSRF 误报。反过来，如果分析器不理解分支条件，就可能简单看到“用户输入流向 HTTP 请求”，然后报出一个可疑路径。

### DFG：值从哪里来到哪里去

DFG 是 Data Flow Graph，数据流图。它表示值之间的传播关系。

你问“这个流是不是 user 输入带来的流动？包含全局变量、数值变量吗？”答案是：**数据流不只指用户输入，它指所有值的传播；用户输入只是安全分析里最关心的一类数据流。**

比如：

```python
name = request.args["name"]
sql = "select * from users where name = '" + name + "'"
db.execute(sql)
```

这里有几条数据流：

```text
request.args["name"] -> name
name -> sql
sql -> db.execute(sql)
```

如果是全局变量，也一样可能形成数据流：

```python
GLOBAL_CMD = request.args["cmd"]
os.system(GLOBAL_CMD)
```

如果是普通数值，也有数据流：

```python
a = 1
b = a + 2
```

只不过漏洞分析通常不会关心所有数据流，而是关心“不可信 source 的数据流”。所以可以这样区分：

| 类型 | 是否是数据流 | 安全分析是否重点关注 |
|---|---|---|
| `a = 1; b = a` | 是 | 通常不重点关注 |
| `name = request.args["name"]` | 是 | 重点关注，因为来自外部输入 |
| `config = GLOBAL_CONFIG` | 是 | 视情况关注，配置可能影响安全逻辑 |
| `token = os.environ["TOKEN"]` | 是 | 可能关注，环境变量可能是敏感数据 |

所以 DFG 的“数据流”不是专指用户输入，而是所有值传播；污点分析是在 DFG 的基础上给部分值贴上“不可信”标签。

### SSA：为什么要把变量拆成单赋值

SSA 是 Static Single Assignment，静态单赋值形式。它的核心是：每个变量版本只赋值一次。

你原来的理解里提到“有些量只改了一次，不会再被修改，所以我们改掉它就可以影响整个逻辑，类似全局变量”。这个理解需要调整一下。SSA 不是为了找“只改一次的关键变量”，而是为了把“同名变量在不同时间点的不同值”拆清楚。

比如原始代码：

```python
x = request.args["name"]
x = sanitize(x)
db.execute(x)
```

人看得懂：第一行的 `x` 是不可信输入，第二行之后的 `x` 可能被净化了。但机器如果只看变量名 `x`，容易混淆。

SSA 会把它拆成类似：

```text
x1 = request.args["name"]
x2 = sanitize(x1)
db.execute(x2)
```

这样分析器就能明确：

```text
x1 是 source
x2 来自 sanitizer(x1)
sink 使用的是 x2，不是 x1
```

SSA 对数据流和污点分析的价值是让赋值关系更清楚，尤其在变量反复赋值、分支合并、循环中非常有用。

分支里还会出现类似 phi 的概念：

```python
if cond:
    x = request.args["name"]
else:
    x = "safe"
db.execute(x)
```

SSA 可能抽象成：

```text
x1 = request.args["name"]
x2 = "safe"
x3 = phi(x1, x2)
db.execute(x3)
```

`phi` 表示 `x3` 可能来自不同分支。这样分析器就知道 sink 处的值可能有一条路径来自用户输入。

### CPG 与 Joern：把多种图合在一起查

CPG 是 Code Property Graph，代码属性图。它通常把 AST、CFG、DFG/PDG 等信息放进同一个图数据库式结构里，方便查询。

Joern 是一个基于 CPG 的代码分析平台，常用于漏洞挖掘和程序分析研究。你可以把 Joern 理解成：

```text
源码 -> CPG -> 用查询语言在图上找漏洞模式
```

CPG 的特点是“统一图查询”。例如同一个查询可以同时关心：

1. AST 上某个函数调用是不是 `execute`。
2. 数据流上参数是否来自用户输入。
3. 控制流上是否经过某个校验分支。

这和 YASA 的 UAST 不是一回事。CPG 强调把多种程序关系放入一个属性图；UAST 强调把多语言 AST 统一成同一种抽象语法结构。二者都服务程序分析，但抽象重心不同。

可以粗略对比：

| 表示 | 核心目标 | 典型代表 | 你该怎么记 |
|---|---|---|---|
| AST | 表示单语言语法结构 | 各语言 parser | 代码长什么样 |
| UAST | 多语言统一语法/语义结构 | YASA-UAST | 不同语言放进统一分析框架 |
| CPG | 合并 AST/CFG/DFG 等多种关系 | Joern | 把代码变成可查询的属性图 |
| SSA | 让变量赋值关系更清楚 | 编译器/分析器 IR | 同名变量拆成不同版本 |

### UAST：YASA 的核心不是“又一个 AST”

UAST 是 Unified Abstract Syntax Tree，统一抽象语法树。它的价值不是“把代码变成树”这么简单，而是把不同语言里语义相近的结构映射到统一节点。

例如：

```python
for x in xs:
    print(x)
```

```javascript
for (const x of xs) {
  console.log(x)
}
```

```go
for _, x := range xs {
    fmt.Println(x)
}
```

三种语言语法不同，但都表达“遍历集合”。UAST 希望把它们映射成统一的迭代结构，方便后续分析逻辑复用。

这就是 YASA 对 YASA-VulnAgent 的关键意义：如果系统要支持 Java、Python、Go、JavaScript，就不能每种语言都从零写一套分析逻辑。UAST 让上层的污点规则、source/sink 模型、Agent 工具返回结构更容易跨语言统一。
我不太理解Unified的意义，直接区分开语言是为了降低工程量吗，每个语言单独分析感觉能够更深层次的囊括每个语言的特点，统一的目的是为了加速吗

不过要注意：UAST 仍然不是漏洞检测结果。更准确的理解是：

```text
UAST 解决“多语言代码如何进入统一分析框架”
数据流/指针/污点分析解决“值如何传播、风险是否形成路径”
查询/checker 解决“我要找哪类漏洞模式”
Agent 解决“如何基于证据判断可利用性并验证”
```

### 表示层之后是否已经形成 codemap

你说“经过这个阶段，我们已经针对代码形成了一个清晰的 codemap”。这个方向是对的，但要更精确一点。

表示层之后，我们通常得到了：

1. 语法结构：函数、类、表达式、调用、赋值。
2. 控制结构：分支、循环、异常、返回。
3. 部分基础关系：调用点、变量使用点、字段访问点。

但完整的“数据从 source 到 sink 的路径”通常还没有完全得到。原因是很多传播关系需要进一步求解：

1. 函数调用需要 call graph。 所以上面获取的赋值关系对于函数还是表面的吗，比如说DFG里面，x1=xxx, x2= fun1(x1),exec(x2)，这里面的fun1没有具体体现，所以需要call graph来展开体现
2. 对象字段需要 points-to/alias。 这个是基于数据流建立更复杂的指针映射的结构吧
3. 动态分派需要解析可能的目标方法。 什么是动态分派，我不理解
4. 跨文件/跨模块需要更大的项目级分析。指针层能够实现跨文件/跨模块识别，因为指针的调用能够将不同的位置链接到一起
5. sanitizer 是否有效需要语义规则。这个语义的解释是什么，谁可以提供语义规则

所以更稳的说法是：**表示层提供 codemap 的底图，数据流/指针/污点分析在底图上计算风险路径。**

## 第二层：指针/别名层回答“这个名字背后到底是谁”

你觉得指针/别名层和表示层都在做“程序可解析建模”，这个感觉很合理。区别在于：

```text
表示层：代码结构是什么
指针/别名层：运行时这些变量、引用、字段可能指向哪些对象
```
可以看出来，一个是静态分析，一个是运行时分析，这些对象的感知是在运行时获取的吗，那能够获取全面吗？
表示层更像“地图上有哪些道路和地点”。指针/别名分析更像“这些路牌指向的实际地点可能是哪几个”。

### 指针分析是不是数据流的一部分

指针分析和数据流分析关系很近，但回答的问题不同。

数据流分析问：

```text
值从 A 是否能传播到 B？
```

指针分析问：

```text
变量/引用 A 可能指向哪些对象？
变量 A 和变量 B 是否可能指向同一个对象？
```

在 Python、JavaScript、Java、Go 里，即使不写 C 语言那种 `int *p`，也到处都是“引用”。

例如：

```python
user_input = request.args["name"]
profile = {}
alias = profile
alias["name"] = user_input
sql = "select * from users where name = '" + profile["name"] + "'"
db.execute(sql)
```

这里 `alias` 和 `profile` 指向同一个字典对象。污点写入的是 `alias["name"]`，读取的是 `profile["name"]`。如果分析器不知道二者是别名，就可能漏掉 SQL 注入路径。

所以你的直觉“指针只是指代关系，它实际上没有创建新的对象”很接近核心。更准确地说：

1. 指针/引用本身是访问对象的名字或通道。
2. 创建新对象的是 allocation site。
3. 多个变量可能指向同一个对象。
4. 污点是否传播，经常取决于这些指向关系。

### Points-to：变量可能指向哪些对象

Points-to relation 是指针分析的核心结果。它不是“明确一定指向哪里”，而是“静态分析能推断出可能指向哪里”。

例如：

```python
if cond:
    x = {"name": request.args["name"]}
else:
    x = {"name": "safe"}

sql = x["name"]
```

静态分析不一定知道 `cond` 运行时真假，所以会得到：

```text
x -> object_created_at_line_2
x -> object_created_at_line_4
```

也就是 `x` 可能指向两个抽象对象。后续污点分析就要考虑 `x["name"]` 可能来自 tainted object，也可能来自 safe object。

这也是静态分析常有误报的原因之一：为了不漏报，它会保守地保留多个可能性。

### Alias：两个变量可能引用同一个对象

Alias 是别名关系。它回答：

```text
a 和 b 是否可能指向同一个对象？
```

例子：

```python
profile = {}
p = profile
p["name"] = request.args["name"]
db.execute("select * from users where name = '" + profile["name"] + "'")
```

这里 `p` 和 `profile` 是 alias。写入 `p["name"]` 会影响读取 `profile["name"]` 的地方。

在漏洞分析里，alias 的影响很直接：

1. 如果 alias 分析太粗，会把无关对象混在一起，造成误报。
2. 如果 alias 分析太弱，会漏掉跨引用传播，造成漏报。

### Abstract object：静态分析里的“运行时对象近似”

Abstract object 不是类，也不一定是具体内存地址。它是静态分析为了近似运行时对象而创建的抽象代表。

为什么需要抽象对象？因为静态分析通常不运行程序，没法知道运行时到底创建了多少对象。例如：

```python
def make_user(name):
    return {"name": name}

u1 = make_user(request.args["name"])
u2 = make_user("safe")
```

运行时可能创建两个字典对象。但静态分析常用“分配点”来代表对象：

```text
dict_object_at_make_user_return
```

如果分析更上下文敏感，它可能区分：

```text
dict_object_at_make_user_called_from_u1
dict_object_at_make_user_called_from_u2
```

所以 abstract object 可以理解为：

```text
运行时对象的静态近似版本
```

它不是类。类是代码里的类型定义；抽象对象是分析器为了追踪对象关系造出来的“代表物”。
所以说抽象对象是静态分析时为了仿真出运行时对象而创建的一个东西，其根据分析的精度区分出不同的抽象对象

### Allocation site：对象创建位置

Allocation site 是对象创建位置。它通常指源码里的创建表达式位置，而不是运行时具体内存地址。

例如：

```python
profile = {}
items = []
user = User()
```

这里可能有三个 allocation site：

```text
line 1: {}
line 2: []
line 3: User()
```

静态分析常用 allocation site 命名 abstract object：

```text
Obj_line_1_dict
Obj_line_2_list
Obj_line_3_User
```

为什么用创建位置命名？因为静态分析没有运行时地址，但源码位置稳定、可回溯、可解释。

要注意一个问题：如果同一行在循环里创建很多对象，allocation site abstraction 可能把这些运行时对象合并成一个抽象对象。

```python
for name in names:
    users.append({"name": name})
```

运行时可能创建很多字典，但静态分析可能用同一个 allocation site 代表它们。这会带来近似，也可能带来误报。
这种不应该也是会有上下文敏感与否的区别之类的吗

### Dynamic dispatch：运行时决定调用哪个方法

Dynamic dispatch 是面向对象语言里的一个核心问题。它的意思是：代码里看到的是一个方法调用，但真正调用哪个实现，要到运行时根据对象类型决定。

例如：

```python
class MysqlDB:
    def execute(self, sql):
        ...

class LogDB:
    def execute(self, msg):
        ...

def run(db, value):
    db.execute(value)
```

`run` 函数里只有 `db.execute(value)`。静态看不出 `db` 是 `MysqlDB` 还是 `LogDB`。如果 `db` 可能是 `MysqlDB`，那 `execute` 可能是 SQL sink；如果 `db` 是 `LogDB`，可能只是日志输出。

points-to 可以帮助分析：

```text
db -> MysqlDB object
db -> LogDB object
```

然后 call graph 才能知道：

```text
run -> MysqlDB.execute
run -> LogDB.execute
```

你问“什么是方法，调度什么方法”。方法就是绑定在类或对象上的函数，比如 `db.execute`。调度就是决定这次调用具体进入哪个实现。

在 Java、Python、JavaScript、Go interface 中，这个问题都很重要。

### Call graph：跨函数数据流的骨架

Call graph 是函数/方法之间的调用关系图。

比如：

```python
def handler(request):
    name = request.args["name"]
    return query_user(name)

def query_user(name):
    sql = "select * from users where name = '" + name + "'"
    return db.execute(sql)
```

Call graph 是：

```text
handler -> query_user
query_user -> db.execute
```

没有 call graph，分析器只能在单个函数里看数据流；有了 call graph，才能追踪：

```text
request.args["name"] -> handler.name -> query_user.name -> sql -> db.execute
```

所以 call graph 是跨函数数据流和污点分析的基础。

### 表示层和指针/别名层的粒度差异

你问“是不是两个层次做的颗粒程度不一样”。可以这样理解：

| 层次 | 主要问题 | 举例 |
|---|---|---|
| 表示层 | 代码结构是什么 | 这里有函数调用、赋值、字段访问、if 分支 |
| 指针/别名层 | 名字和引用背后可能是谁 | `a` 和 `b` 是否同一个对象，`db.execute` 到底是哪一个 execute |
| 数据流/污点层 | 值和风险如何传播 | 用户输入是否经过字段、函数返回、拼接流到 sink |

所以它们不是重复，而是逐步加语义。表示层让代码“可读”；指针/别名层让引用“可解析”；污点层让安全风险“可追踪”。

## 第三层：分析精度术语是在控制误报和漏报

这些 sensitivity 术语看起来抽象，但本质都是在回答一个问题：

```text
分析器要不要区分更多上下文？
```

区分得越多，通常越准，但更慢、更复杂。区分得越少，通常更快，但误报更多，甚至漏报。

### Flow-sensitive：这里的 flow 是控制流顺序

Flow-sensitive 里的 flow 指的是控制流顺序，也就是语句执行先后。

例子：

```python
x = request.args["name"]
x = "safe"
db.execute(x)
```

流敏感分析会看到：

```text
先 x = 用户输入
后 x = safe
sink 使用的是 safe
```

因此可能不报 SQL 注入。

流不敏感分析不区分语句顺序，只看这个函数里出现过：

```text
x 可能来自用户输入
x 被 db.execute 使用
```

因此可能误报。

所以 flow-sensitive 的“流”主要是程序执行顺序，不是“数据流里的流”。它通过控制流顺序提升数据流判断精度。

### Context-sensitive：区分不同调用场景

Context-sensitive 是上下文敏感。这里的上下文不是自然语言里的“前后文语境”，而是函数被谁调用、以什么调用链调用、参数从哪里来。

例子：

```python
def build_sql(name):
    return "select * from users where name = '" + name + "'"

def handler1(request):
    sql = build_sql(request.args["name"])
    db.execute(sql)

def handler2():
    sql = build_sql("safe")
    db.execute(sql)
```

上下文敏感分析会区分：

```text
build_sql called from handler1: name tainted
build_sql called from handler2: name safe
```

上下文不敏感分析可能把两次调用混在一起：

```text
build_sql 的 name 既可能 tainted，也可能 safe
```

这会导致 handler2 也被怀疑。

所以 context-sensitive 不是“调用对象、函数、变量都算上下文”这么宽泛，而是分析器为函数调用建立的区分维度。

### Field-sensitive：区分对象不同字段

Field-sensitive 是字段敏感。它区分对象的不同属性或字段。

例子：

```python
profile = {}
profile["name"] = request.args["name"]
profile["role"] = "user"

db.execute("select * from roles where role = '" + profile["role"] + "'")
```

字段敏感分析会知道：

```text
profile["name"] tainted
profile["role"] safe
```

字段不敏感分析可能把整个 `profile` 当成一个整体：

```text
profile tainted
```

于是 `profile["role"]` 也被误认为 tainted，造成误报。

你的理解“对象的不同属性值”是对的。字段敏感就是不要因为对象里一个字段被污染，就把整个对象所有字段都污染。

### Path-sensitive：区分不同分支路径

Path-sensitive 是路径敏感。这里的路径主要指控制流路径，尤其是 if/else、循环条件、异常分支等。
循环条件的变量数量他是怎么确定的，if/else可以根据每个条件来定，循环条件就得依赖运行时了吧
例子：

```python
url = request.args["url"]
if is_internal_ip(url):
    return "blocked"
requests.get(url)
```

路径敏感分析能理解：

```text
走到 requests.get(url) 的路径上，is_internal_ip(url) 应该为 false
```

如果 `is_internal_ip` 是可靠 sanitizer/filter，那么 SSRF 风险可能下降。

路径不敏感分析可能只看到：

```text
url 来自用户输入
代码里有 requests.get(url)
代码里也有 is_internal_ip(url)
```

但它不一定理解这些语句之间的互斥关系和保护关系。

所以你问“是不是 if else 这种分支路径”，答案是：是，最典型就是 if/else 分支；也包括循环、异常、提前 return 等路径差异。

### Object-sensitive：按 receiver object 区分上下文

Object-sensitive 常用于面向对象程序分析。receiver object 就是方法调用左边的对象。

例如：

```python
db1.execute(sql1)
db2.execute(sql2)
```

这里 `db1` 和 `db2` 是 receiver object。对象敏感分析会用“调用发生在哪个对象上”来区分上下文。

再看一个类例子：

```python
class Repo:
    def save(self, value):
        self.db.execute(value)

safe_repo.save("safe")
unsafe_repo.save(request.args["name"])
```

如果不区分 `safe_repo` 和 `unsafe_repo`，分析器可能把两个调用混在一起。Object-sensitive 分析会尽量区分：

```text
Repo.save called on safe_repo
Repo.save called on unsafe_repo
```

这对 Java、JavaScript、Python 这类有对象/方法分派的语言尤其重要。

### 这些精度维度怎么记

可以用一句话记：

```text
flow 区分先后，context 区分谁调用，field 区分哪个字段，path 区分哪条分支，object 区分哪个接收对象。
```

对应到漏洞误报：

| 精度维度 | 不区分时容易出现的问题 |
|---|---|
| flow | 已经被覆盖成安全值仍然报漏洞 |
| context | 安全调用和危险调用混在一起 |
| field | 一个字段污染导致整个对象污染 |
| path | 互斥分支、保护分支被混在一起 |
| object | 不同对象实例的方法调用被混在一起 |

## 第四层：静态分析不是“激活程序”，而是近似所有可能执行

你问“它是怎么充分激活这个程序，让其流动起来？难道是纯粹的静态审计吗？”

这是一个关键问题。YASA 这类工具的主线是静态分析。静态分析通常不真正运行目标程序，而是通过源码、IR、规则和求解算法推断程序可能发生什么。

它不是“激活程序让它流动起来”，而是：

```text
把程序转换成结构化表示
-> 建立控制流、调用图、数据流、points-to 等关系
-> 在这些关系上求解可能路径
```

这和动态测试不同。

| 方法 | 是否运行程序 | 优点 | 缺点 |
|---|---|---|---|
| 静态分析 | 通常不运行 | 覆盖面大，可扫描很多路径 | 可能误报，对动态特性近似 |
| 动态测试 | 运行程序 | 结果更接近真实行为 | 覆盖有限，需要输入和环境 |
| 符号执行 | 半静态半动态思想 | 可推导路径条件 | 路径爆炸，成本高 |
| LLM 审计 | 不稳定，依赖上下文 | 语义理解强 | 幻觉、不可复现 |

YASA-VulnAgent 的设计是混合路线：

```text
YASA 静态分析给出候选事实和污点路径
LLM 判断可利用性和生成 PoC
Docker 沙箱动态验证 PoC
报告模块输出证据链
```

所以不是只靠静态审计，也不是让 AI 自由猜。静态分析负责“找可能路径”，LLM 负责“基于证据推理”，动态验证负责“证明能不能复现”。

## 第五层：污点分析是在数据流上贴风险标签

你对污点分析的直觉是对的：它就是找“哪些输入可能有问题，哪里会触发风险操作”。只是工程上要把这个过程精确定义成 source、sink、sanitizer、propagator、filter 和 taint path。

### Source：不是所有触发点，而是不可信数据来源

Source 是污点源，不是“一切可以触发的源头”，而是“不可信数据进入程序的位置”。

常见 source：

1. HTTP query 参数：`request.args["name"]`
2. HTTP body/JSON：`request.json["url"]`
3. Header/Cookie：`request.headers["X-Forwarded-For"]`
4. 上传文件内容和文件名。
5. 环境变量、命令行参数。
6. 数据库里曾经由用户写入的字段。
7. 消息队列、RPC、Webhook 输入。

安全分析里要谨慎：有些 source 是直接外部输入，有些是间接外部输入。例如数据库字段如果来自用户注册资料，也可能是二次注入 source。

### Sink：危险操作位置

Sink 是危险操作，不同漏洞类型有不同 sink。

| 漏洞 | 常见 sink |
|---|---|
| SQL 注入 | `db.execute(sql)`、ORM raw query |
| 命令注入 | `os.system(cmd)`、`subprocess(..., shell=True)` |
| SSRF | `requests.get(url)`、HTTP client |
| XSS | HTML/template 输出 |
| 反序列化 | `pickle.loads(data)`、`readObject` |
| 路径穿越 | `open(path)`、文件读取/写入 |

Sink 本身不一定有漏洞。只有当不可信数据以危险方式到达 sink，且缺少有效 sanitizer，才形成可疑路径。

### Sanitizer：不是出现过滤函数就安全

Sanitizer 是净化器或约束器。它降低或消除风险，但不能机械理解成“代码里出现 sanitizer 就安全”。

例如 SQL 注入里，真正强的 sanitizer 通常是参数化查询：

```python
db.execute("select * from users where name = ?", [name])
```

弱的 sanitizer 可能只是替换单引号：

```python
name = name.replace("'", "")
```

这不一定可靠，因为不同数据库、编码、拼接方式、转义语义都可能绕过。

所以 sanitizer 判断要看：

1. 它是否在 source 到 sink 的同一条路径上。
2. 它是否作用于真正进入 sink 的那个值。
3. 它是否适用于当前漏洞类型。
4. 它是否语义足够强。

HTML escape 对 XSS 可能有用，但对 SQL 注入没用。URL allowlist 对 SSRF 可能有用，但对命令注入没用。

### Propagator：让污点继续传播的操作

Propagator 是传播器。它定义“不可信性如何从一个值传到另一个值”。

常见传播：

```python
a = source          # 赋值传播
b = a + "suffix"   # 字符串拼接传播
c = f(a)           # 函数参数/返回传播
obj.x = a          # 字段写入传播
d = obj.x          # 字段读取传播
arr.append(a)      # 容器传播
```

Normal data flow 要求值基本保持一致，比如 `b = a`。Taint tracking 更宽松：只要派生值受 source 影响，也可以传播。

例如：

```python
x = request.args["id"]
y = "user_" + x
```

`y` 不等于 `x`，但 `y` 受 `x` 控制，所以 taint tracking 会认为 `y` tainted。

### Filter：规则级和路径级的筛选

Filter 有两层意思。

第一层是规则级过滤：比如只看 SQLi，只看 Python，只看 Flask handler，只看外部可达入口。

第二层是路径级过滤：比如排除已经被参数化查询处理的路径，排除不可达路径，排除测试代码，排除低可信 source。

Filter 不等于 sanitizer。Sanitizer 改变数据风险；filter 改变“哪些结果要保留或报告”。

### Taint path：真正有价值的是证据链

Taint path 是 source 到 sink 的传播链。对 YASA-VulnAgent 来说，它是 Skill-1 给 Skill-2 的核心证据。

以 SQLi 案例为例：

```python
def search_user(request, db):
    name = request.args["name"]
    sql = "select * from users where name = '" + name + "'"
    return db.execute(sql)
```

路径是：

```text
source: request.args["name"]
  -> name
  -> sql = "select ..." + name
  -> sink: db.execute(sql)
sanitizer: none
cwe: CWE-89
```
这个CWE-89是什么意思

LLM 后续应该基于这种结构化路径推理，而不是凭完整代码自由猜测。

### 污点分析具体怎么找

你问“这一步依然是收集信息吧，确定每个特征，然后串起来，具体怎么找，难道是按照规则静态寻找吗？”

答案是：大体上是静态规则 + 图/关系求解。

可以拆成五步：

1. 解析项目，得到 UAST/IR。
2. 建立程序事实：赋值、调用、字段读写、控制流、调用图、points-to。
3. 根据 source/sink/sanitizer/propagator 规则标记节点和边。
4. 在数据流图或关系数据库中求解 source-to-sink 可达路径。
5. 对结果做过滤、排序、路径解释和误报控制。

简化成关系就是：

```text
Source(a)
Flow(a, b)
Flow(b, c)
Sink(c)
not SanitizedPath(a, c)
=> PotentialVulnerability(a, c)
```

这里的 `Flow` 可以来自赋值、传参、返回值、字段读写、容器操作、框架规则等。

所以污点分析不是“AI 看代码猜”，而是程序分析引擎在结构化事实上求解路径。AI 可以辅助规则生成、路径解释、可利用性判断，但不应该替代引擎求解基础路径。

## 第六层：查询层是在表达“我要找哪种路径”

查询层不是另一个漏洞检测魔法模块。它的作用是让你用比较高层的方式描述漏洞模式。

你可以把它理解成：

```text
程序分析引擎负责建库/建图/求解
查询语言负责表达“我要什么结果”
checker 是某个具体漏洞模式的规则封装
```

### 声明式查询：描述结果，不手写遍历过程

声明式查询 Declarative query 的核心是“描述我要什么”，而不是“手把手告诉机器怎么遍历图”。

比如你写：

```ql
from Source src, Sink sink, TaintPath path
where
  src.isHttpInput() and
  sink.isSqlExecution() and
  path.flowsTo(src, sink) and
  not path.hasSanitizer("parameterized_query")
select sink, path, "Potential SQL injection"
```

你表达的是：

```text
我要找 HTTP 输入流向 SQL 执行，并且路径上没有参数化查询净化的路径。
```

你没有写：

```text
先遍历所有函数
再遍历所有赋值语句
再递归调用图
再处理字段别名
再搜索路径
```

这些细节由引擎和运行时库完成。

这和 SQL 很像。你写：

```sql
select * from users where age > 18
```

你描述结果，不关心数据库具体用哪个索引、怎么扫描、怎么优化执行计划。

### QL：CodeQL 的查询语言

QL 是 CodeQL 使用的查询语言。它看起来有点像 SQL，但本质上更接近面向程序关系的逻辑查询语言，语义里有 Datalog 风格。

CodeQL 的工作方式可以粗略理解为：

```text
源码 -> CodeQL database -> QL 查询 -> 结果/路径
```

YASA-UQL 如果能兼容或接近 CodeQL 语法，那么研究者就可以更容易把 CodeQL 生态里的规则思路迁移到 YASA。

但在当前学习项目里要注意：YASA-UQL 和 YASA-MCP 的具体可用状态、语法细节、API schema 都需要官方材料确认。文章里可以用伪 UQL/伪 QL 学概念，不要把它写成已确认接口。

### Datalog：事实 + 规则 + 反复推导

Datalog 是一种逻辑规则语言，适合表达程序分析里的递归关系。

最小化理解就够了：

```text
Fact：已知事实
Rule：从事实推出新事实
Fixed point：反复推导，直到没有新事实
```

例如已知事实：

```text
Assign(a, b)   # a 的值流向 b
Assign(b, c)   # b 的值流向 c
Source(a)
Sink(c)
```

规则：

```text
Flow(x, y) :- Assign(x, y).
Flow(x, z) :- Flow(x, y), Flow(y, z).
Vuln(x, y) :- Source(x), Sink(y), Flow(x, y).
```

这几条规则的意思是：

1. 如果 `Assign(x, y)`，那么 `x` 流向 `y`。
2. 如果 `x` 流向 `y`，`y` 流向 `z`，那么 `x` 流向 `z`。
3. 如果 source 能流到 sink，那么形成可疑漏洞。

固定点就是不断推：

```text
已知 Assign(a,b) -> 推出 Flow(a,b)
已知 Assign(b,c) -> 推出 Flow(b,c)
已知 Flow(a,b) 和 Flow(b,c) -> 推出 Flow(a,c)
已知 Source(a), Sink(c), Flow(a,c) -> 推出 Vuln(a,c)
```

直到再也推不出新的 `Flow` 或 `Vuln`，就达到 fixed point。

这就是为什么 Datalog 适合数据流、调用图、可达性、points-to、污点传播：这些问题本质上经常是“关系的递归闭包”。

### Predicate：判断关系是否成立的逻辑条件

Predicate 是谓词，可以理解为“一个判断关系是否成立的函数/条件”。

例如：

```text
isSource(x)
isSink(y)
flowsTo(x, y)
hasSanitizer(path)
isSqlExecution(call)
```

你理解“判断是否为 source、sink 的东西叫谓词”是对的。

在查询语言里，predicate 有两个作用：

1. 封装复杂判断，提升复用。
2. 让查询更像描述漏洞模式，而不是堆很多底层细节。

例如：

```ql
predicate isHttpInput(Node n) {
  n matches request.args[...] or
  n matches request.json[...] or
  n matches request.form[...]
}
```

真实语法可能不同，但思想就是把“HTTP 输入”的判断封装起来。

### Fact：程序分析得到的已知关系

Fact 是事实。它来自解析和分析结果。

可能的事实包括：

```text
Function(f)
Call(caller, callee)
Assign(lhs, rhs)
ReadField(obj, field, value)
WriteField(obj, field, value)
Return(func, value)
Argument(call, index, value)
Source(node, kind)
Sink(node, kind)
Sanitizer(call, kind)
PointsTo(var, abstract_object)
Alias(var1, var2)
ControlFlow(stmt1, stmt2)
```

这些事实不是人手写全部，而是由 parser、UAST frontend、分析引擎、框架模型、规则库一起产生。

例如 Flask 框架模型可能知道：

```text
request.args[...] 是 HTTP query source
```

数据库 sink 模型可能知道：

```text
db.execute(sql) 是 SQL execution sink
```

### Rule：从事实推导新事实

Rule 是规则。它把基础事实推导成更高层事实。

例子：

```text
Flow(x, y) :- Assign(x, y).
Flow(x, z) :- Flow(x, y), Flow(y, z).
Tainted(y) :- Source(x), Flow(x, y).
PotentialSQLi(sink) :- Tainted(arg), SqlSink(sink), Argument(sink, 0, arg), not Sanitized(arg).
```

你说“这个应该就是漏洞链推理的环节”基本正确。但要区分：

1. 引擎里的 rule 推理是形式化、可复现的关系推导。
2. LLM 的漏洞可利用性推理是语义判断、解释和验证规划。

YASA-VulnAgent 里最好把这两者分开：

```text
程序分析 rule 负责推出候选路径
LLM 负责解释路径、判断上下文、生成验证
```

### Fixed point：为什么要反复推导

固定点 fixed point 是数据流分析常见求解方式。原因是程序里的关系经常要递归传播。

例如：

```python
a = request.args["name"]
b = a
c = b
d = c
db.execute(d)
```

如果只推一轮，只能得到：

```text
a -> b
b -> c
c -> d
```

要知道 `a -> d`，就需要反复组合已有 flow：

```text
a -> b -> c
a -> c -> d
a -> d
```
这里面的a->c->d应该怎么解释，是删减了a->b->c->d的长链条中的节点吗

直到所有可推出的传播关系都被推出。

### Path query：不仅告诉你有问题，还告诉你路径

Path query 是路径查询。它不仅输出“这里可能有 SQLi”，还输出 source 到 sink 的路径。

这对人工和 LLM 都很重要。

如果只输出：

```text
db.execute(sql): Potential SQL injection
```

你还要自己找 `sql` 来自哪里。

如果输出：

```text
request.args["name"] -> name -> sql -> db.execute(sql)
```

LLM 和人都可以检查：

1. source 是否真是不可信输入？
2. 传播是否真实？
3. 中间是否有 sanitizer？
4. sink 是否真危险？
5. 漏洞是否外部可达？

所以 YASA-VulnAgent 的 Skill-1 不应该只给“告警”，而应该给结构化 path。

## 第七层：引擎、AI、checker 的边界

你问“真正的引擎如何做到解析代码、构建数据库/图/IR、求解路径？我还以为这块求解路径是结合 AI 来搞的，这个引擎是什么？”

这是 YASA-VulnAgent 里最容易混淆的边界。

### 引擎是什么

引擎是执行程序分析的系统组件。以 YASA 语境看，核心大概包括：

```text
UAST frontend parser
point-to analyzer
taint checker
query/checker runtime
```

它做的是：

1. 读取源码。
2. 转成 UAST/IR。
3. 建立程序事实。
4. 跑 points-to、数据流、污点传播。
5. 执行 checker 或查询。
6. 输出路径、位置、证据、错误状态。

这部分应该尽量确定、可复现、可测试。它不是 LLM。

### AI 在哪里

AI 不应该替代引擎完成基础路径求解。更合理的位置是：

1. 帮助理解候选路径是否真的可利用。
2. 结合框架语义和业务上下文解释漏洞。
3. 生成 PoC 或验证步骤。
4. 归纳报告和修复建议。
5. 辅助生成候选 checker，但需要引擎执行验证。

所以链路应该是：

```text
引擎：这里有 source-to-sink 路径
LLM：这条路径在上下文里是否可利用，怎么验证
沙箱：PoC 是否真的复现
```

这也是为什么项目强调 grounding：LLM 的判断必须引用 YASA 返回的结构化事实，而不是凭空猜漏洞。

### 引擎是不是严格遵守 AST/UAST 信息

可以说引擎基于 AST/UAST/IR 和分析规则运行，但不是只机械读取 AST。它还会加入：

1. 框架模型：例如 Flask、Spring、Express 的入口和 source。
2. 库模型：例如数据库 API、HTTP client、命令执行 API。
3. 指针/别名近似：对象关系和动态分派。
4. 控制流近似：分支、异常、循环。
5. 自定义规则：source/sink/sanitizer/propagator。

所以它不是“AST 上简单搜索字符串”。它是在程序结构和规则上做关系求解。

### 信息不可能全包含，所以要承认边界

你担心“没法保证当前信息一定全包含”，这个判断非常重要。

静态分析会遇到很多不确定：

1. 动态导入、反射、运行时代码生成。
2. 动态类型和 monkey patch。
3. 第三方库没有源码或模型。
4. 配置文件、环境变量、路由表不完整。
5. 跨服务、跨语言、消息队列边界。
6. sanitizer 语义不确定。
7. 框架魔法和装饰器行为复杂。

所以静态分析结果通常不是“数学上完全真实的运行轨迹”，而是“在当前模型和规则下的保守近似”。

这就解释了误报和漏报：

| 问题 | 原因 |
|---|---|
| 误报 false positive | 分析器保守合并了不可能同时发生的路径 |
| 漏报 false negative | 模型缺失、规则缺失、动态行为未解析 |

YASA-VulnAgent 的 PoC 验证环节就是为了解决这个问题：静态分析给候选，动态验证筛真实。

### Checker 是规则，不是整个引擎

Checker 可以理解为“一条具体检测规则或检测器”。它不是整个引擎。

关系如下：

```text
Engine：通用分析平台，负责解析、建模、求解、执行规则。
Checker：跑在 Engine 上的一条漏洞检测逻辑。
Query：Checker 的一种表达方式，描述要找的模式。
Result：Checker 执行后得到的告警或路径。
```

比如：

```text
SQLi checker：找 HTTP 输入流向 SQL 执行且无参数化查询的路径
SSRF checker：找用户可控 URL 流向 HTTP client 且无 allowlist 的路径
Command injection checker：找用户输入流向 shell 执行且无白名单的路径
```

所以 checker 更像“规则插件”或“检测器”，engine 是“运行这些规则并提供程序分析能力的平台”。

## 这篇地图对应到 YASA-VulnAgent 怎么学

结合 `~/yasa-vuln-agent` 当前资料，学习优先级可以这样排：

### 第一优先级：把 YASA 的三段架构讲清楚

你至少要能讲清：

```text
YASA-UAST：把多语言代码转成统一表示
YASA-Engine / point-to analyzer：做对象、调用、数据流关系求解
Taint Checker：基于 source/sink/sanitizer/propagator 找污点路径
```

这部分对应申报书里的“为什么不是纯 LLM，也不是传统 SAST”。

### 第二优先级：用 SQLi 案例串起所有概念

不要先追求复杂漏洞。先把一个 SQLi 例子打穿：

```python
def search_user(request, db):
    name = request.args["name"]
    sql = "select * from users where name = '" + name + "'"
    return db.execute(sql)
```

你要能标出：

1. AST：赋值、拼接、函数调用。
2. CFG：函数内顺序执行路径。
3. DFG：`request.args["name"] -> name -> sql -> execute`。
4. Source：`request.args["name"]`。
5. Propagator：赋值、字符串拼接。
6. Sink：`db.execute(sql)`。
7. Sanitizer：无。
8. Query：HTTP input flows to SQL execution without parameterized query。
9. Skill-1 输出：结构化 taint path。
10. Skill-2 输入：路径 + 代码上下文 + CWE-89。

这比抽象背术语更有效。

### 第三优先级：理解精度维度如何影响误报

先不用证明算法，只要能说出每个维度会影响什么误报：

```text
flow-sensitive：变量后续被覆盖或净化时更准
context-sensitive：同一函数多处调用时更准
field-sensitive：对象不同字段不混淆
path-sensitive：if/else 和保护条件不混淆
object-sensitive：不同对象实例的方法调用不混淆
```

这部分能支撑你解释 YASA 的价值：它不是简单模式匹配，而是多维高精度分析。

### 第四优先级：理解查询层和 checker

你要能写出伪查询，不要求一开始能写真实 CodeQL/YASA-UQL。

例如 SQLi：

```ql
from Source src, Sink sink, TaintPath path
where
  src.isHttpInput() and
  sink.isSqlExecution() and
  path.flowsTo(src, sink) and
  not path.hasSanitizer("parameterized_query")
select sink, path, "Potential SQL injection"
```

然后你要能解释：

1. `src.isHttpInput()` 是 source 谓词。
2. `sink.isSqlExecution()` 是 sink 谓词。
3. `path.flowsTo(src, sink)` 是路径可达性。
4. `not path.hasSanitizer(...)` 是排除已净化路径。
5. 这条查询可以封装成 SQLi checker。

### 第五优先级：理解 Agent 层只消费事实，不负责凭空发现

YASA-VulnAgent 的关键不是“LLM 更聪明，所以能找漏洞”。更可信的主张是：

```text
YASA 提供结构化程序事实和污点路径
LLM 在事实约束下判断可利用性
PoC 沙箱验证降低误报
报告模块输出可审计证据链
```

所以后续设计 Skill 时，要始终问：

1. Skill-1 返回的 source、sink、path、sanitizer、代码位置是否清楚？
2. Skill-2 的判断是否引用这些证据？
3. Skill-3 是否能验证而不是只描述？
4. Skill-4 是否把证据链写进报告？

## 最小学习闭环

如果你今天只想抓住最小闭环，可以按这个顺序学：

1. 看懂 `cases/SQLi贯穿案例.md`。
2. 在纸上画出 `request.args["name"] -> name -> sql -> db.execute(sql)`。
3. 标出 AST/CFG/DFG 分别能提供什么信息。
4. 改写例子，加一个对象字段传播：

```python
profile = {}
alias = profile
alias["name"] = request.args["name"]
sql = "select * from users where name = '" + profile["name"] + "'"
db.execute(sql)
```

5. 解释为什么 alias 和 field-sensitive 会影响这条路径。
6. 加一个 sanitizer：

```python
db.execute("select * from users where name = ?", [profile["name"]])
```

7. 解释为什么这时 SQLi 风险下降。
8. 写出一条伪查询。
9. 设计一个 `query_taint_paths` 风格的推断 tool 输出。

示例输出：

```json
{
  "cwe": "CWE-89",
  "source": {
    "expr": "request.args['name']",
    "kind": "http_query_param",
    "location": "app.py:2"
  },
  "sink": {
    "expr": "db.execute(sql)",
    "kind": "sql_execution",
    "location": "app.py:4"
  },
  "path": [
    "request.args['name'] -> name",
    "name -> sql via string concatenation",
    "sql -> db.execute(sql)"
  ],
  "sanitizers": [],
  "confidence": "medium",
  "api_status": "inferred_schema_for_learning"
}
```

注意最后的 `api_status`：当前 YASA-MCP 具体 API 未确认，这种 schema 只能作为学习和规划用的推断设计。

## 最后用一句话收束

YASA-VulnAgent 要学的不是一堆孤立名词，而是一条证据生产链：**UAST 把多语言代码放进统一表示，points-to/alias 让对象关系可解析，数据流和污点分析把 source-to-sink 路径算出来，查询/checker 把漏洞模式表达成可执行规则，Agent 再基于这些结构化事实做可利用性判断、PoC 验证和报告生成。**
