## 核心概念

### 为什么需要子智能体

单一 Agent 跑到底看起来最省事——不需要额外调度、不需要传递上下文、没有协调开销。但在实际工作中，单一 Agent 有两个根本性的瓶颈：

**上下文窗口膨胀（Context Overflow）**。每个工具调用、每次输出、每条错误信息都在往上下文里塞。几轮交互之后，窗口里塞满了中间产物，模型处理后续任务的效率明显下降。不是上下文不够长，而是有效信噪比在持续走低。

**注意力稀释（Attention Dilution）**。模型在一个上下文中同时处理研究、编码、审查、测试、部署——每个阶段都在同一个窗口中叠加。处理后期任务时，前期任务的残留上下文仍在消耗模型的注意力预算。结果不是干不好，而是越到后面越干不好。

用一个具体场景来说明：假设你让一个 Agent 做"调研 Supabase 的 Row Level Security 实现，写一个 Demo 应用，审查代码，然后部署"。前两步还好，到审查和部署阶段，Agent 的上下文里还塞着十几篇调研文档和中间代码草稿，它必须在这些噪音中找到重点。更麻烦的是，如果调研过程中出了一个误解，这个错误会像滚雪球一样污染后续所有步骤。

子智能体的核心思路就是**隔离上下文**：每个子 Agent 拥有独立的上下文窗口，只关注自己负责的那一块，任务完成后把精炼后的结果传回父 Agent。父 Agent 不用关心子任务的中间细节，只收结论。

### 五种使用信号

课程给出了五种明确的信号，说明"是时候派子 Agent 了"。每个信号我配一个实际场景来理解：

**1. 需要大量研究的任务（Heavy Research）**

场景：你要开发一个接入 Stripe Payment Intents API 的功能。Agent 需要先通读 Stripe 文档、理解 Webhook 签名验证机制、查阅 PCI 合规要求、再翻翻社区的坑和最佳实践。

如果这些全挤在主 Agent 上下文里，真正写代码时窗口里堆满了文档摘要。正确的做法：派一个子 Agent 专门做研究，输出一份精炼的"Stripe 集成要点"，主 Agent 拿到这个结论再开始编码。

**2. 多个独立任务（Multiple Independent Tasks）**

场景：你在重构一个项目，需要同时更新 README、迁移数据库 schema、升级依赖版本、改 CI 配置。这四个任务完全不相互依赖。

串行做效率低，并行做但全挤在一个上下文里又会彼此干扰。正确的做法：每个独立任务各派一个子 Agent，并行推进。父 Agent 做最终整合。

**3. 新鲜视角验证（Fresh Perspective）**

场景：你写了一个复杂的数据库迁移脚本，逻辑涉及多层事务嵌套和回滚处理。

人是自己最难发现自己错误的 —— Agent 也一样。同一个 Agent 审查自己的输出，相当于在同一个上下文里自问自答，很难跳脱出原有的思维路径。正确的做法：派一个没有参与过写代码的子 Agent 专门做 Code Review，它带着纯净的上下文去读代码，更容易发现问题。

**4. 事前验证（Pre-validation）**

场景：你要重构一个核心模块，涉及多个文件的修改。

直接动手改，改到一半发现方向不对，白费功夫。正确的做法：先派一个子 Agent 写一份实施计划（Plan），再派另一个子 Agent 独立审查这个计划的可行性——"这个方案有没有遗漏边界情况？依赖更新会不会 break 现有的 API？"

事后修 Bug 的成本远高于事前堵漏洞。Pre-validation 把审查看似多了一步，实际上节省了试错成本。

**5. 流水线工作流（Pipeline Workflow）**

场景：一个标准的"Research → Implement → Review → Deploy"工作流，各个阶段有明确的顺序依赖，且每个阶段需要不同的上下文。

这本质上是多个步骤的串联，天然适合用 Pipeline 模式。每个阶段是独立的子 Agent，上一个的输出就是下一个的输入。

---

除了这五个正式信号，课程还给出了三个更直觉的判断标准，适合快速决策：

| 直觉信号 | 说明 |
|----------|------|
| **多文件（Multiple Files）** | 改动涉及 3 个以上文件，且文件间逻辑相对独立 |
| **独立任务（Independent Tasks）** | 子任务之间没有强耦合，可以单独完成 |
| **我想要另一个观点（I Want Another Opinion）** | 你直觉上觉得当前的 Agent 可能陷入思维定势 |

这三个信号不需要复杂的分析——如果你心里在犹豫"是不是该另起一个 Agent"，那大概率就是该了。

### 四种协作模式

#### 1. Fan-Out / Fan-In 扩散汇聚

```
          ┌──► SubAgent A ──┐
          │                  │
Task ──►  ├──► SubAgent B ──┼──► Aggregate ──► Result
          │                  │
          └──► SubAgent C ──┘
```

**工作方式**：父 Agent 把 N 个独立任务并行派给 N 个子 Agent，各自执行完毕后，由父 Agent（或一个专门的汇总子 Agent）收集所有结果并整合。

**适用场景**：多个任务彼此独立，没有顺序依赖。比如同时查多个 API 文档、同时为多个模块生成单元测试、同时准备多个环境的部署配置。

**优势**：并行执行，整体耗时理论上等于最慢的那个子任务。

**代价**：汇总阶段需要处理来自不同子 Agent 的输出格式差异；如果子任务过多，父 Agent 的上下文在汇总时仍可能被撑满（缓解方案：分批汇总或层级式 Fan-in）。

**与 MapReduce 的类比**：本质上是 MapReduce 在 Agent 上的映射。Map 阶段并行处理独立单元，Reduce 阶段整合结果。

#### 2. Pipeline 管道

```
Step 1 ──► Step 2 ──► Step 3 ──► Final Result
SubAgent A    SubAgent B    SubAgent C
```

**工作方式**：子 Agent 串接成一个链条，每个阶段输出成为下一阶段的输入。数据和上下文沿着管道单向流动。

**适用场景**：任务有明显的阶段性顺序。典型的例子是 Research → Plan → Implement → Review → Deploy。

**优势**：每个阶段都有专注的上下文，不会被前后阶段干扰。管道天然支持在任意节点插入人工审核（Human-in-the-Loop）。

**代价**：整体耗时是所有阶段的总和——任何一个阶段慢了都会阻塞整条线。如果管道的某个阶段出错，下游阶段可能基于错误前置结果继续执行，所以每个阶段建议附加验证。

**与 Fan-Out/Fan-In 的组合**：这两种模式不互斥。典型场景是管道中的某个阶段内部再做 Fan-Out。比如 Plan 阶段输出架构方案后，Implement 阶段内部对多个独立模块做并行实现。

#### 3. Supervisor 监督者

```
                ┌─── Specialist A (研究)
                │
Supervisor ─────┼─── Specialist B (编码)
                │
                ├─── Specialist C (审查)
                │
                └─── Specialist D (部署)
```

**工作方式**：一个父 Agent（Supervisor）不直接执行具体任务，而是调度多个专业化的子 Agent。Supervisor 负责任务拆解、分工指派、进度追踪、结果整合。子 Agent 各司其职，只在自己擅长的领域工作。

**适用场景**：需要多种不同专业能力的复杂任务。比如一个全栈功能开发需要前端、后端、数据库、部署四种能力，很难有一个 Agent 在所有方面都精通。

**优势**：专业化带来质量提升——每个子 Agent 可以加载对应的 skill 和参考文档，在其领域内做到最好。Supervisor 视角更全局，能做出更合理的任务分配决策。

**代价**：Supervisor 本身需要足够的上下文来理解全局任务，它自己的上下文窗口也是有限的。Supervisor 的决策质量直接决定整体成败——如果拆错了任务，后续无法追回。

**与 Pipeline 的对比**：Pipeline 强调阶段的顺序依赖，Supervisor 强调分工和专业化。实践中两者经常混合——Supervisor 拆解任务后，部分子任务内部可能再用 Pipeline 或 Fan-Out。

#### 4. Swarm 群体

```
Approach A ──┐
              │
Approach B ──┼──► Compare & Select ──► Best Result
              │
Approach C ──┘
```

**工作方式**：同一个问题，派多个子 Agent 各自用不同的方法、不同的视角去解决。全体完成后，由父 Agent（或其他评估机制）比较各方案，选出最优解，或者综合各方案的优点生成更好的结果。

**适用场景**：开放性问题、有多种可行方案的问题、需要创新或探索的场景。比如产品命名、架构方案选型、设计评审、性能优化策略。

**优势**：多样性带来鲁棒性——不同的 Agent 可能发现不同的边界情况或优化机会。避免了过早收敛到某个次优方案。

**代价**：成本最高——N 种方案就要 N 倍的 token 消耗和运行时间。对比评估本身也有认知开销。

**什么时候值得用 Swarm**：当问题的"错误成本"足够高时——比如选错了架构方向可能浪费数月时间，花几倍的 token 成本去做多种方案对比就是值得的。

---

四种模式不是互斥的。实际项目中常见的是混合模式：管道中有 Fan-Out、Fan-Out 的某个分支可能跑一个 Swarm、Supervisor 内部可能再启动子管道。选择哪种模式取决于你的**核心约束**是时间（→Fan-Out）、质量（→Swarm）、专业度（→Supervisor），还是阶段依赖（→Pipeline）。

| 模式 | 何时用 | 核心价值 | 主要代价 |
|------|--------|---------|---------|
| Fan-Out/Fan-In | 独立任务并行处理 | 节省时间 | 汇总开销 |
| Pipeline | 有阶段顺序的任务 | 阶段隔离、专注 | 串行耗时 |
| Supervisor | 需要多种专业能力 | 专业化分工 | 依赖 Supervisor 决策质量 |
| Swarm | 开放性问题、多种方案比较 | 多样性、鲁棒性 | 高 token 成本 |

### 何时不该用子智能体

子智能体不是免费的。每次派生子 Agent 都有固定开销：子 Agent 的初始化、上下文加载、任务理解、结果返回。如果这些开销超过了直接做的成本，就不该用。

具体的误用场景：

**简单任务**。改一个拼写错误、更新一行配置、查一个文档——这种任务在现有上下文中一步完成，派子 Agent 属于杀鸡用牛刀。

**紧密耦合的工作**。子任务之间需要频繁沟通或共享大量中间状态。此时强拆成子 Agent 会导致大量的上下文传递开销，反而比不做还慢。

**单文件改动**。课程给出的三个直觉信号之一就是"多文件"——如果只改一个文件，大概率不需要派子 Agent。

**纯粹为了"并行"而并行**。并行听起来高效，但需要任务真的独立才行。如果有隐性依赖没有被识别，并行反而会导致冲突和返工。

判断的一句话原则：**如果你犹豫要不要派子 Agent，通常该派；如果你确定不用派，那一定不用派。**

### 与 Unit 1 Subagent Delegation 的衔接

回顾 Unit 1 中 Agent Skills 规范提到的 **Subagent Delegation** 模式：

> 将复杂任务拆解后，派子 Agent 各自加载相关 skill 在独立会话中执行，避免主对话上下文被污染，也防止多个 skill 的指令互相干扰。

连接到这里，思路就清晰了：

- **Unit 1 定义了"拆什么"**：Skill 是可组合的连贯工作单元。Subagent Delegation 的动机是避免 context pollution 和 skill 指令冲突。
- **Unit 4 定义了"怎么拆和怎么派"**：四种协作模式给出了架构层面的选择——是并行、串行、监督还是群体比较。

两者互为补充。在做 Agent 系统设计时，组合起来的效果是：先识别出需要的 Skill（Unit 1），再选择对应的协作模式（Unit 4），然后派子 Agent 各自加载 Skill 独立执行。

举个例子：一个"分析用户数据并生成报告"的任务，可以先拆出 `data-analysis` 和 `report-generation` 两个 Skill，然后用 Pipeline 模式串联——分析完成后输出结果，报告 Agent 加载它的 Skill 基于结果生成文档。每个子 Agent 只在自己的上下文中加载对应的 Skill，互不干扰。

## 思考与疑问

- Fan-Out/Fan-In 的汇总阶段其实也是一个"小瓶颈"——当 10 个子 Agent 返回结果时，父 Agent 仍然需要处理 10 份输出。这是否意味着 Fan-Out 在大规模并行时需要一个层级式的汇总结构？类似分布式系统中的聚合树（Aggregation Tree）？
- Swarm 模式和 A/B 测试有概念上的相似——都是"多种方案产出后择优"。区别在于 Swarm 是并发评估，A/B 是线上流量隔离。前者适合方案阶段，后者适合上线验证。这个界限如何在实际项目中保持清晰？
- Pipeline 的一个隐含风险：上游出错时，下游基于错误输入产生看似合理实则错误的输出。这是否意味着每个 Pipeline 阶段后都应该加一个验证步骤？验证步骤本身也可能出错，递归下去怎么办？
- "Subagent Delegation + Skill"的组合让我想到微服务架构——每个子 Agent 加载对应 Skill 独立执行，类比微服务的独立部署和独立上下文。这之间的类比到什么程度会失效？Agent 之间的通信（结果传递）目前是父 Agent 汇总，未来会不会有 Agent 之间的直接通信通道？
- 课程说"如果你犹豫是否要用子 Agent，通常该用"——但我担心这个原则在初学阶段会被滥用。有没有一个更定量的判断标准？比如子 Agent 的 overhead 占主任务时间的比例超过多少就不划算？这个恐怕需要大量经验数据才能回答。
