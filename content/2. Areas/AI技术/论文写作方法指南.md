# 论文写作方法指南

## 推荐工作流

遵循以下顺序能较高质量完成论文：

1. **定章节结构** — 先搭好骨架，明确每章要解决什么问题
2. **画图梳理思路** — 用图把整体逻辑跑通，图能暴露逻辑漏洞
3. **写中文注释** — 每章写好对应的中文注释，不着急写英文
4. **精校中文** — 仔细校对中文，确保表达准确、逻辑通顺
5. **翻译为英文** — 基于精校后的中文生成英文
6. **精校英文** — 确保英文地道、术语统一

> 先中文后英文的好处：中文是母语，能更敏锐地发现逻辑和表达问题。

## 论文的核心是论证，不是写作

论文本质上是在证明：

1. 问题存在
2. 方法合理
3. 方法有效
4. 结论可信

如果这条论证链成立，即使语言普通，论文也有价值。
如果论证链不成立，即使文笔华丽，也没有说服力。

## 实验设计四问

在写每个实验前，先明确：

1. **这个实验验证什么假设？**
2. **指标为什么能够支持该假设？**
3. **对比对象为什么合理？**
4. **结果如何解释？**

如果这四个问题答不上来，说明实验逻辑尚未成立。

### 结果不能自动推出原因

实验结果只能直接说明：
- 覆盖率提高了
- 路径数增加了
- 发现了更多漏洞

不能直接说明：
- 理解了协议语义
- 具有更强推理能力
- 掌握了协议本质

**正确写法：**
- "This suggests that the generated seeds are more effective at reaching unexplored states."
- "One possible explanation is that the hierarchical design better preserves protocol context."

**错误写法：**
- "This proves that the model understands protocol semantics."

---

## 创新点必须具体

差的写法：
- "We propose a novel framework."

好的写法：
- "We introduce a two-layer architecture that separates high-frequency execution from low-frequency semantic reasoning."

## Related Work 的目标不是总结，而是定位

每篇相关工作需要回答：
1. 它解决什么问题？
2. 它的核心方法是什么？
3. 它的局限是什么？
4. 本文如何区别于它？

### 推荐结构

- Template-based fuzzing
- Coverage-guided fuzzing
- Stateful fuzzing
- LLM-assisted fuzzing

每一类最后总结共同不足，再自然引出自己的方法。

## 图表先于文字

图表不是文字的附属，而是论文论证的核心。

### 表格设计原则

- 每张表只回答一个问题
- 指标数量不要过多
- 最重要的数字突出显示
- Caption 要说明实验设置和统计方式

### 图设计原则

- 横轴和纵轴必须有明确含义
- 单位要完整
- 字体大小统一
- 图例简洁

### 写作顺序

1. 先确定图表
2. 再分析图表
3. 最后写正文

## 伪代码、图片、表格：三种信息载体

从"论文叙事功能"的角度看，伪代码、图片和表格分别承担三种不同层次的信息表达任务：

* 伪代码：回答"具体是怎么做的"
* 图片：回答"整体是如何工作的"
* 表格：回答"有哪些事实需要结构化展示"

### 三者核心作用对比

| 表达形式 | 主要回答的问题 | 信息层次 | 适合呈现内容 |
| ---: | :--- | :--- | :--- |
| 伪代码 | How exactly?（具体如何实现） | 微观层面 | 算法流程、决策逻辑、关键步骤 |
| 图片 | How overall?（整体如何协同） | 宏观层面 | 系统架构、工作流程、模块关系 |
| 表格 | What facts?（有哪些结构化事实） | 客观事实层面 | 数据、配置、实验结果、对比 |

### 伪代码（Pseudocode）

伪代码用于将算法逻辑形式化，使读者能够明确理解输入、输出、每步处理、决策条件和复杂度。

**适合展示的内容：**
- 算法步骤：搜索策略、训练流程、调度逻辑、变异算法
- 核心决策机制：种子选择、状态更新、奖励计算、优先级调整
- 数学公式的使用位置：公式在正文中解释，伪代码说明调用时机和逻辑

**一句话总结：** 伪代码的作用是把"方法描述"转化为可执行的算法说明。

### 图片（Figures）

图片用于展示系统的整体结构和信息流，使读者快速建立全局理解。

**适合展示的内容：**
- 系统架构图：模块组成、层次结构、数据流
- 工作流程图：输入 → 处理 → 输出
- 时序图：多模块交互顺序
- 状态图：状态及转移关系
- 结果图：折线图、柱状图、CDF 曲线

**一句话总结：** 图片的作用是用视觉方式帮助读者快速理解整体逻辑和模块关系。

### 表格（Tables）

表格用于结构化展示大量离散事实，使数据便于查找和比较。

**适合展示的内容：**
- 实验环境：CPU、RAM、OS、编译器
- Benchmark 信息：Target、Protocol、LOC、Language
- 参数设置：Timeout、Mutation Count、Learning Rate
- 实验结果：Coverage、Paths、Crashes、Throughput
- 消融实验：去掉某模块后的效果变化

**一句话总结：** 表格的作用是高密度、规范化地展示客观事实与实验数据。

### 三者分工的类比理解

| 元素 | 类比 | 功能 |
| --: | :--- | :--- |
| 图片 | 地图 | 告诉读者整体结构 |
| 伪代码 | 菜谱 | 告诉读者具体步骤 |
| 表格 | 数据清单 | 告诉读者客观事实 |

更简洁地说：
- **图片：讲结构**
- **伪代码：讲逻辑**
- **表格：讲数据**

### 各章节中的典型使用位置

| 章节 | 图片 | 伪代码 | 表格 |
| :--- | :--- | :--- | :--- |
| Introduction | 总体框架图 | 通常不用 | 很少使用 |
| Methodology | 系统架构图、流程图 | 核心算法 | 参数说明 |
| Experimental Setup | 很少使用 | 通常不用 | 环境配置、Benchmark 描述 |
| Results | 趋势图和可视化分析 | 通常不用 | 核心结果 |
| Appendix | 更完整的架构图 | 更完整的算法 | 更大的结果表 |

### 信息层次视角

```
整体层（Macro）
└── 图片：系统结构和流程

算法层（Meso）
└── 伪代码：具体逻辑和步骤

事实层（Micro Facts）
└── 表格：参数、数据和结果
```

### 三者之间的交叉引用

图表之间应构成"导航系统"，帮助读者在不同粒度间切换：

- Figure 可能引用 Algorithm（"如图 2 所示的决策逻辑在 Algorithm 1 中详细描述"）
- Table 可能配合 Figure（Figure 展示趋势，Table 给出精确数值）
- 这种交叉引用是论文信息密度的重要体现

### 粒度选择的灵活性

同一系统可以有多张不同粒度的图：
- **Overview Figure**：展示整体结构和高层数据流
- **Detailed Figure**：某个关键子模块的内部实现

这种粒度梯度在系统工程类论文（如 OS、编译器、网络协议）中很常见。

### Caption 的独立价值

Caption 让 figure/table/pseudocode 成为相对独立、可被单独理解的单元。好的 caption 读下来应该能知道这张图在说什么，而不必回头翻正文：
- Figure 的 caption 描述整体结构和关键信息流
- Algorithm 的 caption 描述输入输出和关键步骤
- Table 的 caption 描述数据来源、实验设置和统计方式

### 常见误区

1. **用表格展示流程** — 流程信息更适合图片
2. **用文字详细描述算法而没有伪代码** — 读者难以准确复现
3. **用图片展示大量精确数字** — 数字对比更适合表格
4. **在伪代码中堆砌数学公式** — 公式应在正文解释，伪代码说明调用位置和逻辑

### 顶会论文的常见模式

几乎所有高水平论文都会遵循以下模式：
1. 一张总架构图（Overview Figure）
2. 若干核心算法伪代码（Algorithms）
3. 多个实验结果表（Tables）
4. 若干趋势图（Plots）

## 主动承认局限（Threats to Validity）

明确说明：
- Benchmark 覆盖范围有限
- LLM 输出存在随机性
- 漏洞确认数量有限
- 成本依赖具体模型

主动承认局限通常会提升论文可信度。

---

## 删除没有信息量的句子

例如：
- "This method is effective and practical."
- "The results are encouraging."
- "The framework is comprehensive."

如果删掉后不影响理解，这句话通常没有必要存在。

---

## 用审稿人思维检查论文

审稿人主要关注：
1. 问题是否重要
2. 方法是否新颖
3. 技术是否合理
4. 实验是否充分
5. 结论是否可信
6. 写作是否清晰

在每一章后问自己：
> 如果我是审稿人，我会质疑什么？

---

## 统计严谨性

没有统计支撑，不要说"显著"。

"significant"通常意味着统计显著性。
如果没有进行统计检验，改用：
- noticeable
- substantial
- consistent
- clear

## 投稿前检查清单

### 逻辑
- [ ] 创新点是否明确
- [ ] 每章是否回答一个清晰问题
- [ ] 实验是否支持结论

### 术语
- [ ] 全文术语是否统一
- [ ] 缩写是否首次定义

### 图表
- [ ] 图表编号正确
- [ ] 引用完整
- [ ] Caption 信息充分

### 引文
- [ ] 每个关键结论都有引用
- [ ] BibTeX 信息完整

### 英文
- [ ] 主语明确
- [ ] 避免长难句
- [ ] 删除空洞形容词

### LaTeX
- [ ] 无 overfull hbox
- [ ] 无 unresolved references
- [ ] 无 unused bibliography

## 文献笔记模板

每篇论文记录：
1. **Problem** — 解决了什么问题
2. **Key Idea** — 核心思想是什么
3. **Strengths** — 优点
4. **Limitations** — 局限
5. **Relation to Our Work** — 与本文的关系
6. **Reusable Figures/Tables** — 可复用的图表
7. **Important Citations** — 重要引用
