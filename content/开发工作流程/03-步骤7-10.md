## 步骤7：QA执行测试

**QA → 测试 → bugs/BUG-xxx.md**

测试执行阶段是整个质量保证工作中最"接地气"的环节。前面的设计、评审、代码实现做得再好，如果测试执行不扎实，问题就会漏到生产环境。作为一名博士生，你可能会想："我又不是要去做 Tester，为什么要学这么细？"答案很简单——在研究中，你的代码最终需要能够被复现、被验证、被他人信任。测试执行的能力直接影响你研究成果的可靠性和可复现性。

### 测试执行阶段的基本流程

进入测试执行阶段后，你应该已经准备好了测试用例（来自步骤4的测试用例设计）。现在的工作就是逐一执行这些用例，并记录每一个发现的问题。

执行测试时，建议按照以下顺序进行：

1. **环境确认**：在开始测试前，先确认测试环境已经按照测试用例中的要求搭建完成，包括操作系统版本、依赖库版本、配置参数等。如果环境与预期不符，应该先解决环境问题再开始测试，否则会发现大量"假 bug"。

2. **按用例执行**：严格按照测试用例中描述的步骤操作，不要跳过，也不要根据自己的"经验"擅自增加或减少步骤。如果在执行过程中发现了测试用例本身的问题（比如步骤描述不清楚、前置条件不对），应该记录下来反馈给测试用例设计者，但不要因此停止测试——记录问题，继续执行，记录你实际做了什么。

3. **记录执行结果**：每个测试用例执行完毕后，立即记录执行结果（通过/失败），不要等到一天结束后再回头回忆。失败的用例需要进一步分析，记录到 bug 文件中。

### Bug 文件的规范写法

每一个新发现的 bug 都应该对应一个独立的文件，放在 `bugs/` 目录下，命名格式为 `BUG-xxx.md`（xxx 为序号或 bug ID）。一个规范的 bug 文件应该包含以下几个部分：

**Bug 描述（能复现的步骤）**

这是 bug 文件中最重要的部分。你需要用清晰、无歧义的语言描述如何复现这个 bug。让没有参与代码开发的人也能按照你的描述成功复现问题。

不好的描述：
```
程序出错了
```

好的描述：
```
1. 使用 Python 3.9 运行 `python train.py --config config.yaml`
2. 等待训练进行到第 50 个 epoch
3. 查看输出日志，发现 loss 值显示为 `nan`
```

**实际结果 vs 预期结果**

```
实际结果：训练到第 50 epoch 时，loss 显示为 nan，后续 epoch 的 loss 一直保持 nan
预期结果：loss 应该正常收敛，数值在 0.0~2.0 之间波动
```

**环境信息**

```
操作系统：Ubuntu 22.04
Python 版本：3.9.12
PyTorch 版本：2.0.1
GPU：NVIDIA RTX 3080, CUDA 11.8
相关配置：config.yaml 中 learning_rate=0.01, batch_size=32
```

**截图或日志**

如果问题涉及到 UI 界面、数值输出、错误信息等，尽量附上截图或完整的日志输出。这能大大加快问题定位的速度。对于研究代码来说，日志尤其重要——你可能需要回顾训练过程中的某些中间结果，日志是唯一可靠的记录。

### Bug 严重程度分类

不是所有的 bug 都需要立即修复。你需要对 bug 进行分级，以便决定修复的优先级和方式：

**阻断性（Blocker）**：导致核心功能完全无法使用，测试无法继续进行。这类 bug 必须立即修复。比如训练脚本完全无法启动，或者关键的数据预处理步骤直接报错退出。

**严重（Critical）**：功能存在严重缺陷，但存在变通方法，或者部分模块可以绕过。比如模型可以训练，但验证集上的评估指标完全不输出。这类问题也应该尽快修复，但如果有临时的 workaround，可以先记录下来。

**一般（Normal）**：功能存在缺陷，但不影响核心流程。比如 UI 上的一个显示错误，或者一个不影响训练结果但影响日志可读性的问题。这类 bug 可以和计划中的下一轮开发一起修复。

**建议（Enhancement）**：不是 bug，而是改进建议。比如"建议增加进度条显示"、"建议优化错误提示信息"等。这类问题可以收集起来，在后续迭代中逐步处理。

分类的目的是帮助你合理安排修复工作的优先级。作为个人开发者，你的时间和精力都是有限的，先修阻断性问题，再修严重问题，最后在有时间的时候处理一般问题和改进建议。

## 步骤8：PM分析并修复Bug

**PM → Claude Code 分析 bug 成因 + 修理 bug → BUG 文件补充说明 + compile**

### 为什么 PM 要分析 bug 成因

很多开发者在遇到 bug 时，习惯性地采用"头痛医头"的方式——看到错误信息就改，看到报错就修。这种方式在简单场景下可能有效，但在复杂系统中往往只能治标不治本。你今天修了一个 `nan` 的问题，明天可能又会出现 `inf`；你在这里加了一个 null 检查，那里可能又会有空指针异常。

PM 分析 bug 成因的目的是理解问题的根本原因。表面上看是一个 `nan`，但可能的原因有很多：学习率过大导致梯度爆炸、数据中存在缺失值未被处理、某处计算逻辑有误导致溢出、初始化方法不当等等。只有找到真正的原因，才能给出长期有效的修复方案，而不是反复打补丁。

### 如何利用 Claude Code 分析 bug

Claude Code 非常适合辅助 bug 分析。你不需要让它直接帮你修 bug，而是先让它帮你"看病"。

将你的 bug 文件（或关键部分）发送给 Claude Code，然后提问类似这样的问题：

```
我遇到了一个 bug（详见下方描述）。请分析可能的原因，
列出最可能的 3-5 个成因，并说明如何验证每个成因。

[Bug 描述]
```

Claude Code 会根据你提供的信息进行分析，给出可能的原因假设。这些假设能帮助你快速缩小排查范围。你可以通过查看日志、添加调试打印、执行特定测试来验证每一个假设，最终找到真正的原因。

举一个实际的例子。假设你发现训练过程中 loss 突然变成 nan，Claude Code 可能会给出以下分析：

```
可能原因 1：梯度爆炸（Gradient Exploding）
- 验证方法：检查训练日志中的梯度范数，如果某次迭代梯度范数急剧增大，则确认
- 修复方向：使用梯度裁剪（gradient clipping），或降低学习率

可能原因 2：学习率过大
- 验证方法：将学习率降低 10 倍，重新运行，观察 loss 是否收敛
- 修复方向：调整学习率或使用学习率调度器

可能原因 3：数据中存在 NaN 或 Inf 值
- 验证方法：在数据加载后、进入模型前打印数据统计信息，检查是否有异常值
- 修复方向：数据预处理时增加 NaN 检测和替换逻辑
```

通过这种方式，Claude Code 扮演了一个经验丰富的开发者的角色，帮助你系统性地分析问题，而不是凭直觉乱试。

### 修复后补充说明到 BUG 文件

找到根因并完成修复后，不要忘记回到 bug 文件，补充以下信息：

1. **根因分析**：简要说明导致 bug 的根本原因
2. **修复方案**：描述具体做了什么修改来修复这个 bug
3. **修复者**：记录是谁完成的修复（如果是团队协作，这很重要）
4. **修复时间**：记录修复完成的时间

这样做的目的是为了未来的维护。当其他人（或者未来的你自己）看到这个 bug 文件时，能快速理解这个问题是怎么来的、是怎么解决的。如果未来再次遇到类似问题，这些记录会非常有价值。

### compile 验证：确保修复后代码仍能构建成功

修复完成后，一个容易被忽略但至关重要的步骤是验证代码仍能正常构建。这里的"构建"不仅是指编译通过，还包括所有依赖都能正确加载、配置没有引入新的错误。

对于 Python 项目，简单的验证方式是：

```bash
# 检查语法是否正确
python -m py_compile your_modified_file.py

# 如果有测试，运行环境测试
pytest

# 检查依赖是否满足
pip check
```

对于编译型语言（如 C++、Rust），验证步骤会更严格，需要确保修改没有引入编译错误或链接问题。

这个验证看起来很简单，但它能防止很多"修复了一个 bug 引入了三个新 bug"的情况。作为博士生，你的研究代码通常会经历多次修改，每次修改后做一次快速的构建验证，能大大减少最终整合时的问题。

### 常见的 bug 类型和修复策略

作为个人开发者，你可能会遇到以下几类常见 bug，这里给出一些通用的修复策略：

**逻辑错误**：代码执行流程正确，但计算逻辑有误。这类 bug 通常通过仔细的代码审查和针对性的单元测试来发现和修复。

**边界条件问题**：代码在正常情况下没问题，但在边界值（如空数组、零除、最大值）时出错。修复时需要增加对边界条件的处理，并补充对应的测试用例。

**并发问题**：多线程/多进程环境下，由于竞态条件导致的问题。这类问题通常难以复现，需要通过仔细的线程安全设计来避免。

**资源泄漏**：内存泄漏、文件句柄未关闭、数据库连接未释放等。这类问题在长时间运行的训练脚本中尤其常见。修复后可以通过监控工具或 profiler 来验证问题是否解决。

## 步骤9：QA回归测试

**QA → 回归测试**

### 回归测试的定义

回归测试（Regression Testing）是指重新执行之前已经通过了的测试用例，目的是验证新引入的修改没有破坏现有功能。回归这个词非常形象——它意味着"倒退"，我们要确保修改没有让系统从"能正常工作"的状态退步到"不能正常工作"的状态。

在传统的软件工程中，每次代码变更后都会执行完整的回归测试套件。这需要大量的时间和资源投入。对于个人开发者或小团队来说，完整回归测试的成本可能过高，因此需要掌握一种"轻量级回归测试"的策略。

### 为什么需要回归测试

你可能会有一个自然的疑问：我只是改了一行代码，为什么要去测那么多不相关的东西？答案是软件的复杂性。代码之间存在依赖，修改一处可能会在意想不到的地方产生影响。

举一个研究代码中的例子：你修改了数据加载器（DataLoader）中的一个预处理步骤，目的是修复某个特定的数据格式问题。这个修改确实解决了那个问题，但你不小心删除了一个条件判断，导致另一种数据格式的处理出现了问题。如果你不做回归测试，你可能在论文提交前才发现这个问题——更糟糕的是，审稿人或评审委员会在复现你的实验时发现了这个 bug，这对你的学术声誉会造成更大的损害。

回归测试的核心价值在于"早发现，早修复"。问题发现得越早，修复成本越低，影响范围越小。

### 回归测试的范围

在完整的测试体系中，回归测试的范围通常包括：

1. **本次修复涉及的功能**：直接验证你修复的这个 bug 是否已经解决。
2. **相关联的功能**：测试那些与修复内容有交互的其他功能是否仍然正常。
3. **核心功能**：测试系统的核心功能是否仍然正常工作，确保修复没有动摇系统的根基。
4. **历史 bug 修复区域**：测试过去修复过的 bug 是否仍然正常，确保没有"历史倒退"。

### 个人开发者如何做轻量级回归测试

作为个人开发者，你不需要、也没有精力去做完整回归测试。下面是一种适合个人开发者的轻量级回归测试策略：

**第一步：影响分析**

在每次修复后，先问自己一个问题："这个修改可能会影响到什么？"不需要写完整的分析，但需要在心里有一个判断。比如：

- 你修改了数据加载器，那么受影响的可能是数据管道相关的所有功能
- 你修改了模型的 forward 方法，那么所有调用这个模型的场景都可能受影响
- 你修改了一个工具函数，那么所有使用这个函数的地方都可能受影响

**第二步：选择重点**

根据影响分析的结果，选择最可能受影响的核心场景进行测试。不需要覆盖所有场景，但需要覆盖最重要的：

- 如果你的研究是关于图像分类的，那么分类相关的完整流程（数据加载→模型前向→损失计算→评估指标）应该测
- 如果你修改了一个工具函数，那么至少要在它最关键的几个调用场景中验证它仍然工作

**第三步：记录回归测试结果**

简单记录你测了哪些场景、结果如何。这不是为了给谁看，而是为了你自己的追踪。一个简单的方法是在 bug 修复文件中增加一个"回归测试记录"部分：

```markdown
## 修复后回归测试

### 测试范围
- [测试点 1] 分类模型在修改后的 DataLoader 下是否正常训练
- [测试点 2] 数据增强流水线是否正确处理所有支持的数据格式
- [测试点 3] 批量推理是否正常输出结果

### 测试结果
- [测试点 1] 通过：训练 10 个 epoch，loss 正常收敛
- [测试点 2] 通过：测试了 3 种数据格式，均正常加载
- [测试点 3] 通过：批量推理结果与单张推理结果一致
```

这种轻量级回归测试不需要很长时间（可能 15-30 分钟），但能给你一个相当强的信心：核心功能没有在这次修改中倒退。

## 步骤10：QA产出测试报告

**QA → 测试报告 → test-report.md**

### 测试报告的目的

测试报告是对整个测试工作的总结和归档。它的目的不仅仅是"汇报结果"，更重要的是：

1. **记录质量状态**：让所有相关方（项目经理、开发者、甚至论文评审）都能了解当前代码的质量状态
2. **追踪问题**：汇总所有发现的 bug，便于后续跟踪和追溯
3. **知识沉淀**：为未来的维护和交接提供文档基础
4. **支持决策**：帮助决定是否可以达到发布的条件，或者需要继续修复

对于博士生来说，测试报告还有一个特殊的价值：在论文中，你可以用测试报告来证明你的代码经过了系统性的验证，从而增强研究结果的可信度。

### 测试报告应该包含的内容

一份完整的测试报告应该包含以下部分：

**1. 测试概要**

```
# 测试报告

## 测试概要

- 测试对象：[项目名称] v[版本号]
- 测试时间：2025-11-01 ~ 2025-11-05
- 测试环境：Ubuntu 22.04, Python 3.9, PyTorch 2.0
- 测试范围：数据加载模块、模型训练模块、评估模块
```

**2. 测试执行情况**

记录测试用例的执行情况，包括通过率、覆盖率等关键指标：

```
## 测试执行情况

| 测试模块 | 用例数 | 通过 | 失败 | 通过率 |
|---------|-------|-----|-----|-------|
| 数据加载 | 15 | 14 | 1 | 93.3% |
| 模型训练 | 20 | 20 | 0 | 100% |
| 模型评估 | 10 | 9 | 1 | 90.0% |
| **合计** | **45** | **43** | **2** | **95.6%** |

覆盖的功能点：数据加载、预处理、模型初始化、训练循环、验证、评估指标输出
未覆盖的功能点（由于时间限制）：异常恢复机制、并行数据加载性能测试
```

**3. 发现的 bug 汇总**

列出所有发现的 bug，包括状态（已修复/未修复）：

```
## 发现的 Bug 汇总

| Bug ID | 描述 | 严重程度 | 状态 | 负责人 |
|-------|------|---------|-----|-------|
| BUG-001 | 训练到第 50 epoch 时 loss 变为 nan | 严重 | 已修复 | 张三 |
| BUG-002 | 数据加载器对 PNG 格式支持不完整 | 一般 | 已修复 | 张三 |
| BUG-003 | 评估指标在小数精度上存在浮点误差 | 建议 | 未修复 | - |

### Bug 详情

详见各 bug 文件：/bugs/BUG-001.md, /bugs/BUG-002.md
```

**4. 遗留问题和建议**

记录未修复的问题以及后续的改进建议：

```
## 遗留问题和建议

### 遗留问题
- BUG-003 由于浮点精度问题的修复成本较高，当前采用文档说明的方式处理，
  要求使用者在比较评估结果时设置合理的容差范围（< 1e-6）

### 改进建议
1. 建议增加自动化测试覆盖率，目前为 65%，目标提升至 80%
2. 建议增加长时间训练的稳定性测试，当前仅测试到 100 epoch
3. 建议增加多 GPU 训练的一致性验证
```

### 为什么测试报告很重要

你可能会觉得，作为个人开发者，做这么多文档工作是不是太形式化了？实际上，测试报告的价值远超你的想象：

**对研究者而言**：当你需要复现自己的实验时（几个月后甚至几年后），测试报告能帮助你快速回忆起当时的测试情况。如果你需要向审稿人或评审委员会说明你的代码质量，测试报告是最直接、客观的证据。

**对团队协作而言**：如果你的研究代码需要与他人共享或交接给他人维护，测试报告能帮助他们快速理解系统的质量状态，降低沟通成本。

**对个人成长而言**：写测试报告的过程是一个反思和总结的过程。你会不自觉地思考：测试是否充分？有没有遗漏的场景？这种反思能帮助你不断提升自己的开发能力。

**可复现性的证明**：在科学研究中，可复现性是核心原则之一。测试报告是你对代码质量进行过系统性验证的证明。当你的实验结果受到质疑时，测试报告可以说明你的结果是经过验证的，而不是偶然的。

### 测试报告的维护

测试报告不是一次性文档，而是需要随着项目进展持续更新的活文档。建议你在每次重要的里程碑（如完成一个阶段开发、准备发布版本、提交论文前）都更新一次测试报告。保持它的时效性，这样当你真正需要它的时候，它才是有用的。

另外，建议使用版本控制来管理测试报告，这样可以追踪测试结果随时间的变化趋势。如果某个版本的测试通过率突然下降，你就能快速定位问题是从哪个版本开始引入的。
