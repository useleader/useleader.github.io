2.2.2 排级算法

PageRank

原理：民主表决

$\Pr(t_i)$入度; $|t_i|$出度 $d$影响因子

入度为什么是概率呢？

优点：

  1. 直接高效
  2. 主题集中

缺点：

  1. 忽略网页内容，干扰挖掘结果
  2. 结果范围窄
  3. 影响因子与网页获取数量缺乏科学性

用来表现网页等级的一个标准，0-10，值越高越受欢迎

HITS

Hub页面（枢纽页面）指向高质量链接的网页

Authority页面（权威页面）相关领域高质量网页

枢纽值 Hub Scores = sum of all relevant Authority scores

权威值 Authority Scores = sum of all relevant Hub scores

基本思想：相互增强关系

算法
  1. 构建根集合
  2. 扩展集合Base
  3. 计算扩展集base中所有页面的Hub值和Authority值
  4. 排序，结果输出

优点

  1. 知识范围扩大
  2. 搜索时部分地考虑了页面内容，挖掘结果科学性大大增强

缺点

  1. 效率低，实时性差
  2. 主题漂移
  3. 易被操纵者操纵结果
  4. 结构不稳定

第一章 绪论

1.1 数据库系统概述

数据库是一种计算机辅助管理数据的方法与技术，研究如何有效地组织和存储数据，如何高效地获取和处理数据

1.1.1 数据库的4个基本概念

① 数据 Data

数据是数据库中存储的基本对象

Definition

是描述事物的符号记录，是可识别的、抽象的符号

Categories

  • 结构化

    • 可以使用关系型数据库表示和存储，表现为规范的二维表格形式的数据

    • A 13

  • 半结构化

    • 结构化数据的一种形式，并不符合关系数据模型结构，但包括相关标记来分割语义元素，并对记录和字段进行分层

    • 自描述的结构

    • XML、JSON格式数据

  • 非结构化

    • 文档、图片、视频等
Another definition

​ 使用约定俗成的关键字，对客观事物的数量、属性、状态、位置及其相互关系进行抽象表示，以适合在特定领域中用人工或自然的方式进行保存、传递和处理。

信息

Definition

  • 具有时效性，有一定含义的，有逻辑的、经过加工处理的、对决策有价值的数据流。
  • 信息 = 数据 + 处理
数据与信息
  • 数据是信息的符号表示，或称载体；
  • 信息是数据的内涵，是数据的语义解释；
  • 数据是符号化的信息
  • 信息是语义化的数据
知识

Definition

  • 对消息进行归纳、演绎、比较等手段进行挖掘，使其有价值的部分沉淀下来，并于现存的人类知识体系相结合，有价值的信息就转变为知识。
  • 知识是从定量到定性的过程得以实现的、抽象的、逻辑的东西。
  • 知识在数据与信息之上，因它更接近行动，与决策有关。
数据挖掘
  • 从大量、不完全的、有噪声的、随机的、模糊的实际应用数据中提取有用信息和知识的过程
  • 常用算法
    • 分类、聚类、回归、关联规则
    • 异常检测、特征分析、变化和偏差分析

② 数据库

Definition

长期存储在计算机内、有组织的、可共享的大量数据的集合

基本特征

  • 数据按一定的数据模型 组织、描述和储存
  • 可为各种用户 共享
  • 冗余度较小
  • 数据独立性较高
  • 易扩展

③ 数据库管理系统

Definition

  • 用户与操作系统之间的一层数据管理软件
  • 帮助用户定义、创建、维护、控制数据库访问的软件系统
  • 基础软件

用途

  • 科学地组织和存储数据
  • 高效地获取和维护数据

主要功能

  • 数据定义功能
    • 提供数据定义语言（DDL）
    • 定义数据库中的数据对象
  • 数据组织、存储和管理
    • 分类组织、存储和管理各种数据
    • 确定组织数据的文件结构和存取方式
    • 实现数据之间的联系
    • 提供多种存取方式提高存取效率
  • 数据操纵功能
    • 提供数据操纵语言（DML）
    • 实现对数据库的基本操作：增删改查
  • 数据库的事务管理和运行管理
    • 数据库在建立、运行和维护时，由DBMS统一管理和控制
    • 保证数据的安全性、完整性；支持多用户对数据的并发使用
    • 发生故障后的系统数据恢复（自动、手动）
  • 数据库的建立和维护功能
    • 数据库初始化：初始数据的装载和转换
    • 数据库备份、转储、恢复功能
    • 数据库的重组织
    • 性能监控等
  • 其他功能

④ 数据库系统

Database System, DBS

系统构成

  • 数据库
  • DBMS
  • Application
  • DB Administrator

常见管理信息系统（MIS）架构

  • Client/Server(C/S)架构
    • 通常是2层架构
    • 胖客户端
  • Browser/Server(B/S)架构
    • 典型三层架构
    • 瘦客户端

架构的分层目的

  • Flexibility：灵活
  • Maintainability：可维护
  • Reusability：可重用
  • Scalability：可扩展性

Layer：逻辑功能分层，上下层之间由接口支撑关系

Tier：Layer的物理实现，可独立物理部署，一个Tier可包括多个Layer

数据库系统特点

  1. 数据结构化
    • 数据库的整体结构化是数据库的主要特征之一
    • 数据用数据模型描述，无需应用程序定义
  2. 数据的共享性高，冗余度低且易扩充
  3. 数据独立性高
    • 物理独立性
    • 逻辑独立性
    • 数据独立性由DBMS的二级映像功能保证
  4. 数据由DBMS统一管理和控制
    • 数据控制功能
      • 安全性保护
      • 完整性检查
      • 并发控制
      • 数据库恢复

1.2 数据模型

  • 对现实世界数据特征的抽象
  • 应满足三方面要求
    • 能比较真实地模拟现实世界
    • 容易为人所理解
    • 便于在计算机上实现
  • 是数据库系统的核心和基础

1.2.1 两大类数据模型

数据模型分为两类

  1. 概念模型
    • 信息模型，它是按用户的观点来对数据和信息建模
  2. 逻辑模型和物理模型
    • 逻辑模型是按计算机的观点对数据建模，建立的是具体应用的数据库结构，用于DBMS实现，但不依赖具体的数据库软件厂商和版本。主要包括网状模型、层次模型、关系模型、面向对象数据模型、对象关系数据模型、半结构化数据模型。
    • 物理模型是对数据最底层的抽象，数据在系统内部的表示方法和存取方法，在磁盘或磁带上的存储方式和存取方法。

客观对象的抽象过程-两部抽象

  1. 将客观对象抽象成概念模型
  2. 将概念模型转换为某一数据库管理系统支持的数据模型

数据建模是业务需求理解和概念抽象的过程

抽象过程：理解-区分-命名-表达

数据建模的三个层次/阶段：

  1. 概念模型
  2. 逻辑建模
  3. 物理建模

1.2.2 概念模型

用途

  • 用于信息世界的建模
  • 设计人员和用户之间交流的语言

基本要求

  • 较强的语义表达能力
  • 反应业务逻辑，用客户的述语描述
  • 简单、清晰、易于用户理解

概念模型的一种表示方法：

  • 实体-联系方法
    • 用E-R图来描述现实世界的概念模型

基本概念

  1. 实体（Entity）：可以是具体的或抽象的
  2. 属性（Attribute）：某一特性
  3. 码/键（Key）：唯一标识
  4. 实体型（Entity Type）：用实体名及其属性名集合来抽象和刻画同类实体
  5. 实体集（Entity Set）：同一类型实体的集合
  6. 联系（Relationship）：
    • 现实世界中事物内部及事物之间的联系反应为class内部的联系和class之间的联系
    • class内部的联系通常是指attributes’ relationship
    • class之间的联系通常是指Entity Set Realationship
    • 一对一、一对多、多对多

1.2.3 数据模型的组成要素

  • 数据结构
  • 数据操作
  • 数据的完整性约束条件

数据结构

数据模型的数据结构：描述数据库的组成对象，以及对象之间的联系，是对系统静态特性的描述

数据操作

对数据库中各种对象的实例允许执行的操作的集合，包括操作及有关操作规则

操作类型：

  • 查询 Select
  • 更新
    • 插入 Insert
    • 删除 Delete
    • 修改 Update

数据完整性约束条件

  • 一组完整性规则的集合
  • 完整性规则：给定的数据模型中数据及其联系所具有的制约和依存关系
  • 用以限定符合数据模型的数据库状态及其状态的变化，以保证数据的正确、有效和相容

完整性约束条件的定义：

规定必须遵守的、基本的、通用的完整性约束条件；提供定义完整性约束条件的机制，以反映具体应用所设计的数据必须遵守的特定的语义约束条件

1.2.4 常用的数据模型

网状模型、层次模型、关系模型、面向对象数据模型、对象关系数据模型、半结构化数据模型。

1.2.5 层次模型

树形结构

结点双亲唯一、一对多的实体联系

1.2.6 网状模型

允许一个以上的结点无双亲、一个节点可以有多于一个的双亲

1.2.7 关系模型

逻辑结构是一张二维表

  • 关系 Relation 一个关系对应一张表（包括数据
  • 元组 Tuple 一行
  • 属性 Attribute
  • 主键 Key
  • 域 Domain 相同数据类型的值的集合
  • 分量 Component 元组的一个属性值
  • 关系模式 Relation Schema 对关系的结构描述

关系必须是规范化的，满足一定的规范条件

  • 关系的每一个分量必须是一个不可分的数据项，不允许表中还有表

数据操作是集合操作，操作对象和操作结果都是关系

关系的完整性约束条件：

  1. 实体完整性
  2. 参照完整性
  3. 用户定义的完整性

1.3 数据库系统的结构

从数据库应用开发人员角度看，数据库系统通常采用三级模式结构，是数据库系统内部的系统结构

从数据库最终用户角度来看，结构分为

  • 单用户结构
  • 分布式结构
  • 客户-服务器
  • 浏览器-服务器

模式 Schema

  • 逻辑结构和特征的描述
  • 型的描述
  • 反映的是数据的结构及其联系
  • 模式相对稳定

实例 Instance

反映数据库某一时刻的状态

1.3.2 三级模式结构

模式（逻辑模式）

  • 全体数据的逻辑结构和特征的描述
  • 所有用户的公共数据试图

一个数据库只有一个模式

模式：数据库系统模式结构的中间层

模式的定义

  • 数据的逻辑结构
  • 数据之间的联系
  • 数据有关的安全性、完整性要求

外模式

  • 用户使用的局部数据的逻辑结构和特征的描述
  • 数据库用户的数据视图
  • 介于应用域模式之间

内模式

  • 数据物理结构和存储方式的描述
  • 对用户半透明

1.3.3 数据库的二级映像功能与数据独立性

  • 外模式/模式映像
    • 保证数据的逻辑独立性
  • 模式/内模式映像
